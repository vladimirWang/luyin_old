# 企业微信录音 H5 完整结构文档

本文档用于向 ChatGPT 或其他技术顾问说明当前“企业微信录音 H5”项目的完整结构、运行逻辑、数据流、接口、存储方案和后续可优化方向。

项目名称：`wecom-recorder-h5`

项目目标：做一个可以部署到企业微信中的手机 H5 录音应用。用户从企业微信应用入口打开网页后，可以录音、上传录音、转写文字、管理录音、基于录音内容问答，并能通过时间索引定位到对应音频片段。

## 1. 一句话说明

这是一个 React + Vite 前端、Node.js + Express 后端的企业微信录音 H5 应用。前端负责录音、移动端 UI、录音管理和问答交互；后端负责接收音频、转为 MP3、调用 ASR 转文字、保存 TXT、提供录音问答接口、管理本地 JSON 或 MySQL 数据。

## 2. 现有技术栈

前端：

- React 19
- Vite 6
- lucide-react 图标
- 原生 `MediaRecorder` 录音
- 原生 `navigator.share` 分享
- 原生 `getUserMedia` 麦克风权限

后端：

- Node.js ESM
- Express 5
- Multer 文件上传
- ffmpeg-static 音频转码
- PDFKit 生成问答分享 PDF
- mysql2 可选 MySQL 存储
- 本地 JSON 作为默认轻量数据库

外部服务：

- MiMo ASR：录音转文字
- MiMo-V2.5-Pro：录音内容问答
- OpenAI 转写接口：可选
- 自定义 ASR API：可选

## 4. 当前页面结构

底部有三个主入口：

- 记录
- 录音
- 问答

### 4.1 录音页

组件：`RecorderView`

功能：

- 显示音频波浪线。
- 显示录音时长。
- 点击中间麦克风开始录音。
- 再点一次结束录音。
- 录音中红圈会根据音量波动。
- 停止后把音频上传服务器。
- 上传成功后在记录页生成录音卡片。

录音核心逻辑：

- 使用 `navigator.mediaDevices.getUserMedia({ audio })` 获取麦克风。
- 使用 `MediaRecorder` 录音。
- `RECORDING_DATA_SLICE_MS = 1000`，每 1 秒切一个录音数据片。
- `RECORDING_ROLLOVER_MS = 8 * 60 * 1000`，每 8 分钟做一次内部保护分段，避免长录音到 70 分钟左右被浏览器自动断掉。
- 如果页面隐藏、电话中断、系统暂停麦克风，会尽量请求当前数据片并自动保存。
- 当前修复后的手动逻辑是：点击开始，再点结束；上传失败也回到初始状态，不再进入“继续录音后再结束上传”的状态。

注意：

- 手机真实录音需要 HTTPS。
- 企业微信 WebView 对麦克风权限依赖系统和企业微信版本。
- 来电、锁屏、系统回收 WebView 时，H5 无法 100% 阻止录音中断，只能尽量保存已录片段。

### 4.2 记录页

组件：

- `RecordsView`
- `RecordCard`
- `RecordPreviewOverlay`

功能：

- 显示录音卡片。
- 搜索录音编号、名称、转写内容。
- 上传录音或视频文件。
- 创建文件夹。
- 按文件夹归类。
- 收藏录音。
- 删除到回收站。
- 回收站彻底删除。
- 卡片名称可编辑。
- 标记可编辑。
- 文件夹可重命名和删除。
- 两指缩放卡片布局，类似相册视图，最小可到一行四个。
- 今日录音和历史录音通过卡片边缘或背景微弱差异区分。
- 点击卡片会出现悬浮预览层。

卡片预览层：

- 显示完整转写内容。
- 底部有音频播放器。
- 点击某一段文字可以跳转到对应音频时间。
- 可分享文字 TXT、录音 MP3 或文字 + 录音。
- 可进入该录音的问答页。

### 4.3 问答页

组件：`ChatDetailView`

功能：

- 可以对全部录音提问。
- 可以选择单个录音提问。
- 可以多选多个录音一起提问。
- 可以查看历史聊天记录。
- 历史记录可以折叠到左侧。
- 历史记录可以收藏、删除、分享。
- 收藏夹和历史记录分开显示。
- 输入框支持文字输入、语音输入、图片、拍照、上传录音、上传文件、上传地址。
- 语音输入是按住说话，放开后转文字填入输入框。
- 回答基于录音转写内容。
- 回答后按观点展示，每个观点下面跟对应录音时间索引。
- 一个观点如果对应多个录音时间点，默认折叠，点击可展开。
- 点击时间索引可以播放对应录音片段，再点停止。

问答范围逻辑：

- 如果一开始选择了两个录音一起提问，该对话历史会关联这两个录音。
- 后续查看该历史时，只有相关录音范围内的卡片会显示这条对话。
- 如果一开始只选中一个录音提问，该历史只和这个录音关联。

功能：

## 6. HTTPS 和企业微信

手机端企业微信打开网页录音时，需要 HTTPS。

原因：

- 浏览器安全策略要求 `getUserMedia` 只能在安全上下文中使用。
- `localhost` 和 `127.0.0.1` 是开发例外。
- 手机访问局域网 IP，例如 `http://192.168.1.148`，不是安全上下文，所以麦克风会被禁止。

生产推荐：

```env
HOST=0.0.0.0
PORT=8787
HTTPS_PORT=6443
HTTPS_KEY_PATH=C:\path\to\domain.key
HTTPS_CERT_PATH=C:\path\to\domain.pem
```

企业微信应用入口配置为：

```text
https://mir.hyp-arch.com:6443/
```

## 7. 录音数据流

### 7.1 手机录音流程

```text
用户点击麦克风
  -> H5 请求麦克风权限
  -> MediaRecorder 开始录音
  -> 前端每秒收集 Blob 数据片
  -> 用户再次点击麦克风
  -> MediaRecorder stop
  -> 前端把片段合并为 FormData
  -> POST /api/recordings/segments
  -> 后端合并音频并转为 MP3
  -> 写入 server/storage/audio
  -> 创建 recording 元数据
  -> 异步启动转写任务
  -> 前端记录页显示新卡片
```

### 7.2 上传已有文件流程

```text
用户选择文件
  -> 前端过滤 audio/video 文件
  -> 支持一次选择多个文件
  -> 每个文件并行 POST /api/recordings
  -> 后端统一转 MP3
  -> 异步转写
  -> 记录页生成多个卡片
```

支持格式：

```text
mp3, m4a, wav, webm, aac, mp4, mov, m4v
```

### 7.3 转写流程

```text
录音上传成功
  -> runTranscriptionJob(recordingId)
  -> transcribeRecording(recording)
  -> 根据环境变量选择 provider
  -> MiMo / OpenAI / Custom ASR / local fallback
  -> 得到 transcript segments
  -> 写入 server/storage/transcripts/{id}.txt
  -> 写入 transcript_segments
  -> 更新 recordings.status = ready
```

## 8. 音频存储方案

当前策略：

- 不管上传源文件是什么格式，后端都转成 MP3。
- MP3 默认参数：

```env
STORAGE_MP3_BITRATE=96k
STORAGE_MP3_SAMPLE_RATE=16000
```

本地文件：

```text
server/storage/audio/{recordingId}.mp3
server/storage/transcripts/{recordingId}.txt
```

生产建议：

- 当前本地磁盘适合测试或单机部署。
- 正式多人使用建议迁移到对象存储，例如 OSS、COS、S3。
- 数据库中 `storage_provider` 和 `storage_key` 字段已经预留。

## 12. ASR 和问答逻辑

### 12.1 ASR provider 自动选择

优先级：

```text
ASR_PROVIDER 显式配置
  -> OPENAI_API_KEY 存在时用 openai
  -> MIMO_API_KEY 存在时用 mimo_api
  -> ASR_API_URL 存在时用 custom
  -> 否则 local-fallback
```

### 12.2 MiMo ASR

MiMo ASR 流程：

```text
MP3 或其他音频
  -> 如果文件太大，按 MIMO_CHUNK_SECONDS 切块
  -> 每块转成 16k 单声道 MP3
  -> base64
  -> 调用 MiMo chat/completions 的 input_audio
  -> 解析 text 或 segments
  -> 生成 startMs/endMs/text/speakerKey
```

注意：

- 如果 MiMo 返回没有准确时间戳，系统会按文本长度和音频时长估算分段时间。
- 如果 MiMo 失败，当前逻辑会记录错误，已有转写不会轻易丢失。

### 12.3 说话人逻辑

当前说话人依赖 ASR 返回的 speaker 信息。

如果 ASR 返回：

- `speaker`
- `speakerKey`
- `speaker_label`
- `speaker_id`
- `channel`

系统会归一化为 `speaker-1`、`speaker-2` 等。

如果 ASR 没有说话人区分，则默认只有“说话人 1”。

前端支持编辑说话人名字，保存到 `speakerMap`。

### 12.4 问答逻辑

问答先构建录音语料：

```text
[录音 003｜电梯｜01:07-01:15] 文本内容
```

然后调用 LLM：

- 要求基于录音转写回答。
- 不要整段复述原文。
- 回答要有结论和依据。
- 每条依据要带录音编号、录音名、时间点。

本地仍会做 citation 匹配：

- 根据问题和回答文本匹配最相关 transcript segment。
- 每个观点下面展示一个或多个时间点。
- 多个时间点默认折叠。

## 15. 当前结构上的主要问题

这些是后续可以问 ChatGPT 的重点。

### 15.1 前端 App.jsx 过大

当前 `src/App.jsx` 集中了：

- 录音逻辑
- 记录页
- 卡片
- 预览弹层
- 问答页
- 历史记录
- 分享逻辑
- API 调用
- 录音状态机

问题：

- 文件太大，维护困难。
- 组件职责不清晰。
- 状态分散在一个组件中。
- 很难单独测试录音状态机和问答逻辑。

建议拆分：

```text
src/
  api/
    client.js
    recordings.js
    folders.js
    qa.js
  hooks/
    useRecorder.js
    useRecordings.js
    useQaChat.js
    useShare.js
  components/
    Recorder/
    Records/
    Chat/
    Settings/
    Common/
  utils/
    audio.js
    format.js
    files.js
```

### 15.2 后端任务没有真正队列化

当前上传后调用 `runTranscriptionJob(id)`，属于进程内异步。

问题：

- Node 进程重启会丢失正在转写的任务状态。
- 多个大文件并行转写时可能压垮进程。
- 任务状态不够完整。

建议：

- 引入任务队列，例如 BullMQ + Redis。
- 转写任务独立 worker。
- 前端轮询或 WebSocket 获取任务状态。
- `transcription_jobs` 表真正投入使用。

### 15.3 文件存储仍是本地磁盘

问题：

- 多服务器部署时无法共享音频文件。
- 备份和扩容困难。
- 静态文件权限管理较弱。

建议：

- 使用对象存储。
- 数据库只保存 object key。
- 音频播放使用签名 URL。
- 转写 TXT 也可以存对象存储。

### 15.4 MySQL 写入方式还比较粗

当前 `saveMysqlDb(db)` 会先删除多张表再重新插入。

问题：

- 数据量大后性能差。
- 并发写入有风险。
- 不适合正式生产。

建议：

- 改成 Repository 层。
- 每个 API 只更新相关表。
- 使用事务但不全量重写。
- 增加 migration 工具。

### 15.5 缺少企业微信身份体系

当前用户是默认用户。

问题：

- 没有真正区分企业微信成员。
- 所有人可能看到同一套录音。
- 缺少权限控制。

建议：

- 接入企业微信 OAuth。
- 获取 `code` 后换取 userId。
- 建立 user/session。
- 每条 recording 绑定真实 user_id。
- 加企业/部门权限。

### 15.6 录音可靠性受 H5 限制

问题：

- 电话、锁屏、后台、系统资源回收都可能中断录音。
- H5 无法完全保证“任何程序都不中断”。

建议：

- 接入企业微信 JS-SDK，如果有可用录音能力则作为兜底。
- 或开发原生 App / 小程序插件。
- 长录音改成分段实时上传。
- 每段上传成功后服务器合并。

### 15.7 说话人分离能力依赖 ASR

当前没有本地说话人分离模型。

问题：

- 如果 MiMo ASR 不返回 speaker，系统只能显示一个说话人。

建议：

- 选择支持 diarization 的 ASR 服务。
- 或接入 pyannote / speaker diarization 服务。
- 说话人重命名映射独立存储。

### 15.8 问答索引仍是启发式匹配

问题：

- LLM 回答观点与 transcript segment 的对应可能不完全准确。
- 多处引用需要更强的检索算法。

建议：

- 增加 embedding 向量检索。
- 按 segment 建向量索引。
- 回答时先检索 topK segment，再让 LLM 基于这些片段回答。
- citations 使用 segment id，而不是事后文本匹配。

## 17. 推荐的生产级目标架构

```text
手机企业微信 H5
  -> HTTPS 反向代理 Nginx
  -> Node.js API Server
      -> Auth Service 企业微信身份
      -> Recording Service 录音元数据
      -> Upload Service 分片上传
      -> Storage Adapter OSS/COS/S3
      -> Job Queue Redis/BullMQ
      -> ASR Worker
      -> QA Service
      -> Vector Search Service
  -> MySQL 元数据
  -> Object Storage 音频和 TXT
  -> Vector DB 转写片段向量
```

推荐分阶段：

第一阶段：

- 保持现有功能。
- 拆分前端组件和 hooks。
- 后端拆 controller/service。
- 修复 MySQL 增量写入。

第二阶段：

- 引入对象存储。
- 引入任务队列。
- 转写 worker 独立。
- 前端显示任务进度。

第三阶段：

- 企业微信 OAuth。
- 多用户权限。
- 向量检索问答。
- 更稳定的分片上传和断点恢复。

第四阶段：

- 完整日志监控。
- 错误追踪。
- 文件生命周期管理。
- 数据备份和审计。

## 18. 当前项目的核心价值

当前版本已经能完整演示企业微信录音产品闭环：

```text
打开企业微信应用
  -> H5 录音
  -> 上传服务器
  -> 转 MP3
  -> ASR 转写
  -> TXT 存档
  -> 卡片管理

  -> 分享导出
```

它适合作为 MVP 和原型验证版本。下一步如果要真正生产使用，最应该优先解决的是：身份权限、任务队列、对象存储、MySQL 增量写入、前端模块拆分和问答检索准确性。
